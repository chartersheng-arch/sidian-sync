# BUGrecord 反馈归纳方法论

> 用于将多个用户提交的 BUGrecord.md 归纳成高正确率分析流程
> **执行者**: skill作者（你）
> **输入**: 多个BUGrecord.md
> **输出**: 融入skill的高正确率分析流程

---

## 收集标准

### 最少样本量

| 场景 | 最少BUGrecord数 | 说明 |
|------|-----------------|------|
| 单一问题类型（如仅"无声"） | 5份 | 足以发现共性偏离模式 |
| 多问题类型混合 | 8份 | 需覆盖不同类型 |
| 紧急迭代 | 3份 | 快速验证后先上线 |

### 有效记录标准

一份有效的BUGrecord应包含：
- ✅ 明确记录了初始错误方向
- ✅ 有触发纠正的契机说明
- ✅ 有偏离根因分类
- ✅ 有优化后的分析流程建议

## 归纳流程

### Phase 1: 原始数据整理

#### 步骤1.1 - 按问题类型分类

```
BUGrecord_pool/
├── silent/          # 无声问题
│   ├── bugrecord_001.md
│   ├── bugrecord_002.md
│   └── ...
├── distortion/      # 杂音/破音
├── latency/         # 延迟问题
├── dropout/         # 断续/卡顿
└── other/           # 其他问题
```

#### 步骤1.2 - 提取偏离轨迹

对每份BUGrecord，提取以下信息到结构化表格：

| BUG ID | 问题类型 | 初始方向 | 偏离点 | 纠正契机 | 根因分类 | 最终正确方向 |
|--------|----------|----------|--------|----------|----------|--------------|
| #001 | 无声 | 查HAL配置 | 只看HAL，忽略路由 | 用户追问设备切换场景 | 层级遗漏 | AudioPolicy路由 |

---

### Phase 2: 模式识别

#### 2.1 识别高频偏离模式

统计每个"偏离根因分类"的出现频次：

```
偏离模式频次统计:
├── 层级遗漏          ████████░░  8次  (40%)
├── 假设验证跳过      █████░░░░░  5次  (25%)
├── 平台特性不熟悉    ███░░░░░░░  3次  (15%)
├── 时序忽略          ██░░░░░░░░  2次  (10%)
└── 其他              ██░░░░░░░░  2次  (10%)
```

#### 2.2 识别偏离链

对高频偏离模式，分析其典型"触发链"：

**层级遗漏的典型触发链**:
```
App层无报错
  → 误判为HAL问题  ← 偏离点1
  → 只查HAL配置
  → 未发现路由异常
  → 绕路很久
  → 用户追问："最近有没有插拔耳机"
  → 发现AudioPolicy路由问题
```

#### 2.3 识别关键判断点

关键判断点是**能提前发现偏离**的检查点：

| 关键判断点 | 如果检查这个 | 就能避免 |
|-----------|-------------|----------|
| "先确认App层有没有AudioTrack创建成功" | App层log | 层级遗漏 |
| "先确认问题发生在哪个场景（通话/音乐/通知）" | 场景信息 | 方向走偏 |
| "先对比正常log的差异点" | 对比分析 | 假设验证跳过 |

---

### Phase 3: 高正确率流程归纳

#### 3.1 构建"必须先做"的检查清单

对于高频偏离模式，归纳出**分析初期必须执行的检查**：

**针对"层级遗漏" → 必须先做分层确认**:
```
☑️ 阶段0. 信息收集时，必须确认以下分层信息：
  [ ] App层：AudioTrack/AudioRecord是否创建成功
  [ ] Framework层：AudioFlinger是否有活跃track
  [ ] HAL层：write/read返回值和错误码
  [ ] Kernel层：DMA/Clock状态

  → 如果任何一层缺失信息，主动询问用户补充
```

**针对"假设验证跳过" → 必须先做假设过滤**:
```
☑️ 阶段1. 自动分析前，必须列出3个可能假设
  [ ] 假设1：<最可能的方向>
  [ ] 假设2：<次可能的方向>
  [ ] 假设3：<容易被忽略的方向>

  → 每个假设必须有对应的验证方法
```

**针对"时序忽略" → 必须先做时间轴对齐**:
```
☑️ 阶段0. 信息收集时：
  [ ] 确认问题发生的精确时间点
  [ ] 提取问题前后2秒的关键日志
  [ ] 按时间顺序排列各层日志

  → 时间轴对齐是分析的第一步，不是最后一步
```

#### 3.2 更新failure-patterns.md

将归纳出的高正确率流程融入到 `failure-patterns.md`：

**添加位置**: 各类问题的排查决策树

**添加内容类型**:

1. **新增决策分支**:
   ```
   无声
   ├── 检查App层AudioTrack状态  ← 新增：防止层级遗漏
   │   └── 创建成功 → 继续检查Framework层
   │   └── 创建失败 → 直接定位App层问题
   └── ...
   ```

2. **新增警告标注**:
   ```
   ⚠️ 常见偏离: 不要在未确认App层状态前直接检查HAL
   ```

3. **新增判断点检查表**:
   ```
   ### 无声问题 - 必须检查的关键判断点

   | 检查顺序 | 检查项 | 正确状态 | 异常意味着 |
   |----------|--------|----------|------------|
   | 1 | AudioTrack创建 | SUCCESS | App问题 |
   | 2 | Mixer thread有track | yes | Framework问题 |
   | 3 | HAL write返回 | >=0 | HAL问题 |
   | 4 | DMA interrupt | 收到 | Kernel问题 |
   ```

#### 3.3 更新skill.md

如果偏离模式具有跨问题类型的共性，更新核心方法论：

**可能需要强化的铁律**:
```
⚠️ 铁律：未经完整调查，禁止给出修复结论
     ↓ 新增
⚠️ 铁律：未经分层确认，禁止跳层分析
```

---

### Phase 4: 验证与迭代

#### 4.1 小规模验证

选择2-3个历史案例，用新归纳的流程重新分析：
- 如果新流程能更快定位根因 → 验证通过
- 如果新流程仍然绕路 → 分析原因，补充遗漏判断点

#### 4.2 更新skill版本

| 更新类型 | 版本号变化 | 说明 |
|----------|------------|------|
| 小修复（新增判断点） | 2.0 → 2.1 | bugfix |
| 中等更新（新增流程） | 2.0 → 3.0 | feature |
| 大改（重构方法论） | 2.0 → 4.0 | breaking |
