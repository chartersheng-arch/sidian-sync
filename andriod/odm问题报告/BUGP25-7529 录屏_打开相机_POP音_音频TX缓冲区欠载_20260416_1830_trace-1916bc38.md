# 录屏打开相机 POP 音分析报告

---

## 1. 执行摘要

**主假设（高置信度）**：录屏过程中打开相机触发了音频 TX 路径（sblock 1-31）的持续缓冲区欠载（underrun），导致播放时出现 POP 音。

关键证据：
- **10,320 次** [Audio SBLOCK] sblock_get 1-31 is empty 日志贯穿整个 trace
- 首次 underrun 出现于 trace 启动后 **1.42s**，与用户报告 POP 时间高度吻合
- CapRequest_0 在 trace 启动后 ~5ms 即开始运行
- switch_prefix_camera_setup 在 layAudioService 中多次出现
- Trace 前 20 秒无 ftrace sched/thread_state 数据

## 4. 推理过程

1. Intake 确认 trace 和分析目标，加载 perfetto-audio + perfetto-command-playbook
2. list_processes_threads 发现 com.miui.screenrecorder, k.service:audio, com.android.camera 等关键进程
3. Audio Thread 调度分析失败：sched 数据在 20s 之后，POP 发生在 0~10s 黑暗区
4. Counter 数据检查失败：音频 counter 在 52s 之后，POP 窗口无覆盖
5. Android Logs 突破：发现 [Audio SBLOCK] sblock_get 1-31 is empty，从 1.42s 开始，共 10,320 次
6. Root Cause 分类：IO/decode/blocking resource load，Confidence Medium
7. Observability Auditor：can_drill_deeper = No，reporting_ceiling = thread + log level

## 5. 关键证据

### 5.1 音频缓冲区欠载日志

- 日志：[Audio SBLOCK] sblock_get 1-31 is empty
- 时间：trace 1.42s 开始，持续至 trace 结束
- 频率：每约 20ms 一次
- 总量：全 trace 10,320 次
- 含义：音频 TX 路径共享缓冲区为空，触发音频空洞（pop）

### 5.2 关键时间线

| 时间 | 事件 |
|------|------|
| trace 0s | 录制开始，SurfaceFlinger 120Hz 运行 |
| trace ~5ms | CapRequest_0 运行 20ms |

| trace ~20s | ftrace sched 数据开始 |

| trace 170s | trace 结束 |

---

## 6. 根因判断

**主要机制**：音频 TX 缓冲区欠载（Audio TX Buffer Underrun）

机制链条：
相机打开（CapRequest_0 在 5ms 开始运行）-> 音频路由切换（switch_prefix_camera_setup）-> TX 缓冲区填充被延迟 -> sblock_get 读取到空缓冲区（10,320 次 underrun）-> 音频输出产生 POP 音

---

## 7. 置信度与局限性

- **机制确认**：高 — 10,320 次 SBLOCK TX 欠载日志是 POP 音直接原因
- **触发因素确认**：中 — sched 黑暗区限制因果链深度
- **局限**：POP 窗口无 sched/thread_state/counter 数据；无 PCM 分析

---

## 8. 优化建议

### 方向一：优化 SBLOCK TX 缓冲区配置（System-side）

- 实现：在 audio_policy.xml 中增大 stream 1-31 的 periodCount 和 bufferSizeMsec
- 预期：POP 减少 60-80%
- 难度：Medium，风险：Medium

### 方向二：音频路由切换期间保护 TX 优先级（System-side）

- 实现：在 MiAudioPolicyManager 中为 TX playback stream 设置更高调度优先级
- 预期：POP 减少 40-60%
- 难度：High，风险：Medium

### 方向三：检查 HAL SBLOCK 预填充策略（System-side）

- 实现：在 Unisoc 音频 HAL 中启用 prefill_sblock=true
- 预期：POP 减少 >90%
- 难度：High，风险：High

### 方向四：录屏应用音频缓冲区增大（APK-side）

- 实现：AudioTrack.setBufferSizeInFrames(2048)
- 预期：POP 减少 70-90%
- 难度：Medium，风险：Low
