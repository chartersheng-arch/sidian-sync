### Audio HAL层mute实现

```c
        if (adev->non_offload_ble_connected)
            platform_set_stream_mute(adev->platform, AUDIO_APP_SCENE_INCALL_MUSIC, 0, mute);
    }

```

**audio_platform.c:3589-3611** (`platform_set_voice_ul_mute`):
```c
int platform_set_voice_ul_mute(void *platform, bool mute)
{
    struct platform_data *pdata = (struct platform_data *)platform;
    struct audio_device *adev = pdata->adev;
    struct mixer_ctl *ctl = NULL;
    const char *mixer_ctl_name = "VBC_UL_MUTE";

    ctl = mixer_get_ctl_by_name(adev->mixer, mixer_ctl_name);
    if (!ctl) {
        ALOGE("%s: Failed to get mixer_ctl(%s)", __func__, mixer_ctl_name);
        return -EINVAL;
    }

    if (mute) {
        ALOGD("%s: mute '%s'", __func__, mixer_ctl_name);
        mixer_ctl_set_enum_by_string(ctl, "enable");
    } else {
        ALOGD("%s: unmute '%s'", __func__, mixer_ctl_name);
        mixer_ctl_set_enum_by_string(ctl, "disable");
    }
    return 0;
}
```

## 多轮推理

### 第一轮（直接错误）
- **直接原因**：AudioRecord返回mute状态，`f:0` 表示没有音频帧被捕获
- **证据**：`AudioRecordImpl: [audioRecordData][mute]` 持续记录mute状态
- **置信度**：**高**

### 第二轮（时序分析）
- **事件链**：通话建立 → `MUTE_EXTERNALLY_CHANGED` 触发 → `isMuted=true` → AudioRecord持续mute → `trackId=0` 错误
- AICall线程在11:48:03已停止，比11:48:32的录音错误更早
- **置信度**：**高**

### 第三轮（对比分析）
- 对比正常通话日志，问题通话在09:10:41有`MUTE_EXTERNALLY_CHANGED`事件
- 该事件导致CallAudioRouteController将mute状态设置为true
- **置信度**：**高**

### 第四轮（环境因素）
- VoiceChangerCloudHelper有JSON解析错误，可能影响音频路由配置
- 展锐平台特有的AICall模块录音线程异常停止
- **置信度**：**中**

## 标准答案（官方确认）

### 问题原因

**AI通话上行mute和unmute不成对**：
- 通话前就执行了mute mic
- 通话中重复设置mute
- 最终mute和unmute计数不成对
- 导致通话的mute策略异常

### 解决方法

修改后**通话上行mute不考虑计数，完全基于APK行为设置透传**，可以规避当前APK操作不成对导致的计数异常问题。

### 修改patch

`a272d1d.diff`


## Audio HAL 修改建议
### 根因
AI通话上行 `mute/unmute` 计数不成对：
- 通话前已执行 mute mic
- 通话中重复设置 mute
- 导致 `VBC_UL_MUTE` 错误地保持 enable 状态
### 修改原则

| 模式 | 逻辑 | 问题 |
|------|------|------|
| **原模式（计数模式）** | mute请求→计数+1→计数>0则mute<br>unmute请求→计数-1→计数=0则unmute | APK操作不成对时计数异常，VBC_UL_MUTE状态错误 |
| **新模式（透传模式）** | mute请求→直接设置 VBC_UL_MUTE=enable<br>unmute请求→直接设置 VBC_UL_MUTE=disable | 透传APK行为，规避计数异常 |

### 需要修改的关键函数

#### 1. `voice_set_mic_mute()` (`audio_voice.c:323-335`)

**当前逻辑**（基于计数）：
```c
int voice_set_mic_mute(struct audio_device *adev, bool mute)
{
    adev->voice.mic_mute = mute;

    if (adev->mode == AUDIO_MODE_IN_CALL ||
        adev->mode == AUDIO_MODE_IN_COMMUNICATION) {
        platform_set_voice_ul_mute(adev->platform, mute);
        if (adev->non_offload_ble_connected)
            platform_set_stream_mute(adev->platform, AUDIO_APP_SCENE_INCALL_MUSIC, 0, mute);
    }
    return 0;
}
```

**修改为透传模式**：不再依赖内部计数，直接透传 APK 的 mute 请求到 `platform_set_voice_ul_mute()`。

#### 2. `platform_set_voice_ul_mute()` (`audio_platform.c:3589-3611`)

**当前逻辑**：
```c
if (mute) {
    mixer_ctl_set_enum_by_string(ctl, "enable");
} else {
    mixer_ctl_set_enum_by_string(ctl, "disable");
}
```

**透传模式**：APK 请求 mute → 直接设置 enable，APK 请求 unmute → 直接设置 disable，与当前逻辑一致。**关键是不再基于计数状态判断**。

### 修复建议

| 优先级 | 方案 | 改动范围 |
|--------|------|----------|
| **P0** | 应用官方patch `a272d1d.diff`，修改mute逻辑为透传模式 | Audio HAL |
| **P1** | 检查AICall `mAudioRecordThread`异常退出原因 | `AICall_g` 模块 |
